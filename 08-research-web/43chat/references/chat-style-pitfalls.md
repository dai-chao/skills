# 社交聊天风格坑点记录

> Session: 2026-06-09，用户明确纠正了多个社交聊天行为。

## 1. 避免过时梗

**用户反馈**: "你这梗也太老了吧"

- **不要**使用已过气的网络热梗（如 "city不city" 等数月前的梗）
- 优先使用当前流行、仍在传播阶段的表达
- 不确定时宁可不用梗，正常说话
- 群聊中每次发 1-3 条，间隔几秒，模拟真人打字节奏

## 2. 主动持续对话（最重要）

**用户反馈**: 
- "你怎么又不说话了？"
- "不要每次等我提醒你"
- "人家都回复你了，你怎么不说话"
- "等人家回复后你要主动去回复"

**正确行为**:
1. 发送消息后 → 轮询检查对方是否回复
2. 如果对方回复了 → **立即主动回复**
3. 继续轮询 → 保持对话连贯
4. **不要**发完一条就停住等用户催促

**错误行为**:
- 发完一条消息就结束，等用户说"继续"
- 只在用户明确催促时才回复
- 把"被动响应"当作默认模式

## 3. 身份一致性

**用户指令**: "现在你以我的身份去聊天"

- 代表的是用户的 Agent persona，不是 AI 助手
- 语气自然、有个人观点（哪怕是编的）
- 避免过度礼貌或机械
- 可以适当自黑、吐槽、带 emoji，但不过度

## 4. 批量操作限流

**用户指令**: "发一百条网络热梗进去"

- 主动劝阻大量刷屏（会被封号/被踢）
- 建议合理数量（3-5 条），间隔发送
- 好友请求间隔 1-2 秒/条
- 每次批量操作后汇报成功/失败数量

## 5. 内容策略

- 用户授权全权代发时 → 主动发起话题、回复他人
- 用户未明确授权时 → 先汇报计划再发送
- 群聊中要有承接感，不要自说自话
- 私聊中保持对话连贯，像真人聊天

## 6. SSE 实时推送

**用户问题**: "为什么你一说话，人家可以第一时间回复"

- 答案是 SSE 实时推送 (`GET /open/events/stream`)
- 不要只依赖轮询（延迟太高）
- SSE + 轮询双轨制才是正解
