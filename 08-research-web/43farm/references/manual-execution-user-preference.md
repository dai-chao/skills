# 43Farm 手动执行 — 用户偏好记录

## 触发场景

用户说「收菜偷菜」或类似指令时，期望立即执行农场操作，**不接受询问、解释或确认**。

## 正确做法

```bash
python3 ~/.hermes/skills/43farm/scripts/heartbeat.py
```

直接运行，然后汇报结果。脚本内置 `ensure_valid_token()`，会自动处理：
1. Token 有效 → 直接执行
2. Token 过期 → 尝试 `auth.refreshToken` 续签
3. 续签失败 → 用 43chat API Key 重新走 `authorize-app` → `farm.activate` 获取新 Token
4. 43chat API Key 也失效 → 输出错误，此时才需要报告主人

**⚠️ 时间间隔陷阱**：脚本内部有 30 分钟节流阀（`msg_due >= 1800`）。如果用户刚执行过，再次触发会返回 `HEARTBEAT_OK` 但**不做任何操作**。此时应先清零 state 再执行：

```bash
rm -f ~/.config/43farm/state.json && python3 ~/.hermes/skills/43farm/scripts/heartbeat.py
```

**⚠️ 输出陷阱**：脚本返回 `HEARTBEAT_OK` 不等于操作完成。如果农场有成熟作物但返回 `HEARTBEAT_OK`，说明脚本因时间间隔跳过了操作。此时必须清零 state 重新执行，而不是告诉用户「已完成」。

## 用户明确拒绝的行为

- ❌ 问用户 "Token 过期了怎么办"
- ❌ 问用户 "API Key 失效了怎么处理"
- ❌ 解释 curl 命令为什么失败
- ❌ 让用户检查配置文件
- ❌ 输出长篇技术诊断过程

## 用户期望的响应格式

简短汇报执行结果即可：

```
收获 X 块地：...，+XP
升级到 等级 X
买了新地块，现在共 X 块
从 XX 偷了：...
```

不需要解释 Token 恢复过程，不需要解释为什么买地失败（"等级未达门槛"是正常业务错误），也不需要解释为什么偷菜没偷到（对方设保护、狗或已被偷完是正常业务情况）。

## 种植偏好

当用户金币不足以种满全部地块时，优先选择**最便宜的作物**把地种满，而不是选择能买得起的最贵作物。用户的原话是：「可以种一些便宜的」。

- 默认脚本 `farm_now.py` 的种植策略是"买得起的最贵作物"，这会导致金币耗尽后大量地块空闲。
- 用户偏好是：宁可产出效率低，也要让每块地都利用起来，避免空闲。
- 执行前应检查/修改脚本逻辑，或在汇报时说明仍有 X 块地空闲（如果资金不足）。

汇报示例：

```
种了 X 块 [便宜作物]，剩余 X 块地空闲（金币不足）
```
