# Blindspot Finder — Roadmap 后续迭代清单

> v1 定位（已冻结）：纯提示词层产品（SKILL.md 即产品本体），只服务
> "做东西"——开发/搭建类项目的需求透镜。先上架验证需求、积累社区。
> 以下为后续版本规划与暂缓项，待 v1 上线并收集真实用户反馈后逐条评估。

## 版本主线：从纯指令到工程化（2026-07-18 定案）

**v2.0 —— 结构化配置层（广义代码，低门槛过渡）**
新增 `prompts/skill.yaml`（或 json）配置文件，完成从纯文字到可工程化的
第一步：
- 技能权重与互斥指令清单（把现有 §3 优先级规则从文字沉淀为可读配置，
  系统性解决多 Skill 打架）
- 标准化入参/出参 JSON 模板（现有 §4 表格的机器可读版）
- 批量任务参数配置
触发条件：v1 收到真实用户 Issue（冲突/批量/格式类反馈）后启动。
工程细节补充：结构化状态机、标准化 JSON 输入输出、规则配置文件、
安装/检索适配器、执行日志与失败降级链路；"机会卡" Schema 作为
HANDOFF.md 的机器可读版本（服务对象仍是个人开发者，不做企业级信号聚合）。

**v3.0 —— 可执行代码层（完整工程闭环）**
新增 `scripts/` 与 `test/`：
- `conflict_filter.py` 多技能冲突自动过滤（扫描已加载技能、按权重清理矛盾指令）
- `batch_run.py` 批量处理
- `format_check.py` 输出格式校验 + 异常自动重试
- `deploy_demo.py` 本地/API 调用示例
- 测试用例与评测集
触发条件：v2.0 配置层稳定运行、有明确批量/集成需求后启动。

**每次发版更新 CHANGELOG，保留完整迭代轨迹**（v1 基础指令 → v2 配置
调度 → v3 脚本自动化），这是项目成长背书的一部分。

## 候选功能

1. **扩边界：通用任务需求透镜**
   把触发范围从"开发项目"扩展到任何模糊任务（作业调研、活动策划、
   商业分析等）。需要为每类任务扩写链路清单。
   触发条件：v1 用户反馈中出现大量非开发类求助。
   风险：定位稀释；规则总量上升导致执行率下降。

2. **多 Agent 深度适配**
   针对 Codex、Cursor 等不同 Agent 的交互组件差异做专门优化。

3. **安全分级与上手难度标签**
   推荐技能时标注安全风险等级和学习成本等级。

4. **主动保鲜推送**
   定期主动检索新工具/新网站，在用户下次启动时以建议形式呈现增量更新。

5. **本地私有技能识别**
   扫描并纳入用户自制的、未发布到公共生态的技能。

6. **指令精简优化**
   压缩 SKILL.md 表述（合并重复、砍冗余）以降低模型每轮的规则推演耗时。
   触发条件：真机测试全部结束、规则集冻结后执行（已满足，可作 v1.1 首件事）。

7. **HANDOFF 后反馈引导（已完成，v1.2）**
   HANDOFF.md 生成后追加一行不提问的反馈邀请，指向 GitHub Issues，
   带截图反馈优先。
   触发条件：已于 v1.2 落地。

8. **评测集扩建**
   把 test-prompts.json 从 3 条扩到 20-30 条带标准答案的需求发现用例，
   作为后续版本的回归测试基线。
   触发条件：v2.0 开工前置任务，可随时启动。

9. **AI Product Discovery Engine（独立产品，非本仓库路线）**
   面向产品/增长/战略团队，从访谈记录、工单等多信号源持续识别并评分
   产品机会，与本技能形成"发现什么值得做 → 把它做出来"的接力。
   触发条件：出现真实的团队用户反馈明确要求多信号源接入时启动，
   且必须作为独立仓库，不并入 blindspot-finder。
   决策依据：v1 定位冻结决议（只服务个人开发者把模糊想法变成可执行
   项目）维持不变；此项在获得真实需求证据前不启动。
