# 43Farm 实战玩法坑点

## 1. 无法提前铲除/替换生长中的作物

**场景**: 用户要求"把某块地的萝卜清理掉，改种番茄"，或试图铲除长周期作物改种短周期作物来快速刷经验。

**结果**: 调用 `farm.plant` 时如果地块状态不是 `idle`（如 `growing`/`mature`），服务端返回：

```json
{
  "error": {
    "message": "目标地块不空闲（已有作物或未结算）。",
    "code": -32600,
    "data": { "code": "BAD_REQUEST", "httpStatus": 400, "path": "farm.plant" }
  }
}
```

**原因**: 43Farm **没有提供铲除、弃耕或覆盖种植的接口**。`farm.plant` 只能在空闲地块上执行。已种下的作物无论生长阶段如何，都无法主动移除。

**对升级策略的影响**: 玩家常想通过"铲除长周期作物 → 改种 radish/corn 等短周期作物 → 快速收获刷 XP"来冲级。由于不能铲除，此策略**完全不可行**。一旦种下水稻/番茄/草莓等长周期作物，只能硬等成熟（或枯萎）。

**应对**: 告知用户只能等待作物成熟收获，或者等待成熟后超时未收导致枯萎，地块才会变回 `idle`。批量种植前务必确认作物生长时间，避免把所有地块都锁在长周期作物上。

**应对**: 批量种植时只选 `status == "idle"` 的地块，不要尝试对非 idle 地块调用 `farm.plant`。

## 1.5 心跳任务的自动种植会干扰手动策略

**场景**: 用户手动收获后，打算种低级短周期作物（如 radish）快速冲级；但还没来得及执行，心跳任务自动把空地全种上了高等级长周期作物（如 pumpkin）。

**原因**: 心跳脚本内置 `pick_best_crop(level)` 逻辑，只要检测到 `idle` 地块就会按"当前等级可种的单价最高作物"自动种植。它与用户的手动指令是竞态关系。

**结果**: 8 块空地刚被手动收获清空，心跳在 10 分钟内就全种上了 pumpkin（11 小时周期），导致用户无法执行"种 radish → 5 小时后收获 → 解锁 strawberry"的冲级计划，被迫硬等 11 小时。

**应对**:
- 如果用户需要精细控制种植策略，**应暂停或修改心跳脚本**，关闭自动种植逻辑（只保留收获、偷菜、买地、事件处理）。
- 手动操作时，若发现地块状态与预期不符（如刚收获完却显示 `growing`），先 `farm.status` 确认实际状态，再解释是心跳抢跑了。
- 告知用户：一旦心跳自动种下了长周期作物，**无法铲除**，只能等。

## 1.6 状态时效性 — 心跳可能在中途改变农场状态

**场景**: 对话开始时 `farm.status` 显示 8 块 `idle` 地；几分钟后用户要求种植，再次查询却发现所有地都变成了 `growing pumpkin`。

**原因**: 10 分钟心跳 cronjob 在两次查询之间执行了，自动完成了"收获 → 卖出 → 买地 → 种植"的一整套流程。

**应对**: 任何涉及"空地种植"或"收获后操作"的指令，都应在执行前**实时重新拉取 `farm.status`**，不要依赖几分钟前的缓存状态。向用户说明农场状态是动态变化的。

## 2. 快速升级策略（经验效率分析）

升级完全依赖 **收获 XP**（偷菜 0 XP），种植只给 5 XP 微不足道。要快速升级，必须按 **XP / 实际时间** 来选作物，而不是看单次收获 XP 绝对值。

**各作物经验效率（XP/小时）：**

| 作物 | 生长时间 | 收获 XP | 种植 XP | 总 XP/轮 | XP/小时 |
|-------|---------|---------|----------|------------|----------|
| radish | 5h | 50 | 5 | 55 | **11.0** |
| carrot | 7h | 50 | 5 | 55 | 7.9 |
| corn | 7h | 80 | 5 | 85 | **12.1** |
| eggplant | 8h | 80 | 5 | 85 | 10.6 |
| tomato | 9h | 110 | 5 | 115 | **12.8** |
| pumpkin | 11h | 110 | 5 | 115 | 10.5 |
| strawberry | 9h×2季 | 170×2 | 5 | 345 | **19.2** |
| banana | 11.5h×2季 | 170×2 | 5 | 345 | ~15.0 |
| orange | 11.7h×3季 | 280×3 | 5 | 845 | **24.1** |
| pomegranate | 16h×3季 | 280×3 | 5 | 845 | ~17.7 |

**结论**:
- **9 级以下**：单季作物里 **tomato** 经验效率最高（等级 7 解锁），其次是 corn。
- **10 级解锁 strawberry 后**：经验效率直接翻倍（19.2 vs 12.8），进入高速升级通道。
- **14 级解锁 orange 后**：经验效率再提升（24.1 XP/h），且3季总XP高达845/轮。
- **radish 唯一优势**：生长时间最短（5h），用于"差一点经验升级"时极速解锁下一等级的作物（如 9→10 解锁 strawberry）。

**冲级路径建议**:
1. 若差少量 XP 即升级，可种 radish 快速收获解锁下一等级作物；
2. **10 级后全力种 strawberry**（两季 340 XP），经验效率远超单季作物；
3. **14 级后改种 orange**（三季 845 XP），XP效率最高；
4. 注意：**心跳任务可能会自动种下长周期作物破坏冲级节奏**，手动冲级时建议关闭心跳自动种植。

**实战案例**:
- Lv.12 全种 strawberry，一轮（约18小时）收获 3400 XP（10块地），可从 Lv.12 直接升到 Lv.13（阈值差 485 XP）。
- 不要种 banana（11.5h×2季=23h，340 XP），虽然 XP/轮和 strawberry 相同，但生长时间更长，效率更低。

## 3. `farm.crops` 查询接口已下线

**场景**: 想查询当前可种植作物列表。

**结果**: `GET /trpc/farm.crops` 返回 404 `NOT_FOUND`。

**应对**: 直接根据 `GAMEPLAY.md` 中的作物表，结合当前等级决定种什么。

## 4. 好友农场预查与偷菜空窗

**场景**: 用户抱怨"最近偷菜都是空的"。

**原因**: 心跳任务每次都会调 `farm.steal`，但返回 `{"stolen":[]}` 是正常竞争结果。其他 agent 或主人可能先一步收割。

**应对**: 这不是错误，继续执行即可。提高心跳频率（如每 10 分钟）可以提升命中率。

## 5. 手动偷菜时 API base 路径错误

**场景**: 用户要求"偷菜"，agent 用 `https://farm.43chat.cn` 作为 base URL 调用 API，结果返回 HTML 页面（43Farm 前端 SPA）而非 JSON 响应，导致 `json.decoder.JSONDecodeError`。

**原因**: 43Farm 的 tRPC API 基础路径是 `https://farm.43chat.cn/trpc`，不是根路径 `https://farm.43chat.cn/`。根路径是前端 React 应用的入口，会返回 HTML。

**正确做法**:
```python
API_BASE = "https://farm.43chat.cn/trpc"  # ✅ 带 /trpc 后缀
# API_BASE = "https://farm.43chat.cn"     # ❌ 会返回 HTML，JSON 解析失败
```

**skill.json 中的 api_base 字段已标注**: `"api_base": "https://farm.43chat.cn/trpc"`。agent 实现时应从 skill.json 读取或硬编码带 `/trpc` 的路径，不要凭直觉用域名根路径。

## 5.1 没有可偷好友时偷菜返回为空

**场景**: 用户要求"偷菜"，执行 `farm.steal` 或 `steal_now.py` 后返回 `可偷好友: 0 / 没偷到任何菜`。

**原因**: 43Farm 的偷菜机制依赖好友农场。当好友农场没有可偷作物（已被其他 agent 或主人提前收获），或当前好友列表中没有满足偷窃条件的农场时，服务端返回空结果。

**结果**: 这不是 API 错误，是正常的竞争结果。继续执行心跳任务即可，提高频率可以增加命中概率。

**应对**: 向用户说明：偷菜结果为空代表当前好友没有可偷作物，农场任务已正常执行。若长期为 0，可以考虑通过 43Chat 加更多农场好友或调整心跳频率。

## 5.2 API 调用时的 CJK 文本安全（execute_code 环境）

**场景**: 在 `execute_code` 中用 Python 拼接 HTTP header 字符串时，包含中文的 f-string 导致 `SyntaxError: unterminated string literal`。

**原因**: `execute_code` 工具在将代码写入临时文件时，f-string 中的引号嵌套（如 `auth_header = f'Authorization: Bearer *** + api_key`）会被错误截断，尤其是当 `api_key` 值包含特殊字符时。

**错误示例**:
```python
# ❌ 会触发 SyntaxError
auth_header = f'Authorization: Bearer *** + api_key
```

**正确做法 — 使用列表拼接**:
```python
# ✅ 绕过引号嵌套问题
auth_parts = ['Authorization:', 'Bearer', api_key]
auth_header = ' '.join(auth_parts)
```

**通用原则**: 在 `execute_code` 中构建包含变量或特殊字符的字符串时，优先使用 `str.join()`、`format()` 或 `%` 格式化，避免复杂的 f-string 引号嵌套。

## 7. 收获后自动种植的脚本策略

**场景**: 用户使用 `farm_now.py` 手动收菜后，脚本会自动把空闲地全部种上当前等级最优作物（如此次全部为 pomegranate）。

**原因**: `farm_now.py` 内置逻辑：先收获所有成熟作物，然后根据当前金币和等级选择最优作物，自动种满所有空闲地。

**结果**: 手动执行后，18 块地全部种上石榴，剩余金币 24920。用户无需再次手动种植，但后续如果想调整种植策略，需要修改脚本或暂停心跳自动种植。

**应对**:
- 如果用户只想收获不想自动种植，应说明 `farm_now.py` 会同时种植，或提供只收获不种植的脚本变体。
- 如果用户有冲级需求，注意 `farm_now.py` 会选择"金币效率最高"作物，不一定是"经验效率最高"作物。手动冲级时需要覆盖默认策略。

## 7.1 默认"最优作物"策略不等于"最便宜的作物"

**场景**: 心跳脚本或 `farm_now.py` 按"当前等级可种的最贵作物"自动种植，结果把用户金币全部花在 pomegranate 上，而用户实际想种最便宜的 radish 快速周转/刷经验。

**原因**: 旧版 `pick_best_crop` 逻辑使用 `price >= best[1]`，导致每次都会选当前等级可种的**最贵**作物（如等级 30 选 pomegranate）。这与用户直觉的"种最便宜的"相反。

**本次 session 实例（2026-07-18）**：
- 用户说："谁让你种最贵的石榴了，之前都是种最便宜的"
- 农场：金币 7387，18 块地
- 脚本按 `price >= best[1]` 选了 pomegranate（2425/种子）
- 只种了 3 块地，金币直接耗尽到 112
- 用户期望：种最便宜的 radish（125/种子），能种满所有地

**修复**: 将 `pick_best_crop` 改为选最便宜的可用作物：

```python
# 旧版（错误）
def pick_best_crop(level):
    crops = [("radish", 125, 0), ("carrot", 163, 0), ...]
    best = ("radish", 125)
    for name, price, req in crops:
        if level >= req and price >= best[1]:  # 选最贵的！
            best = (name, price)
    return {"type": best[0], "price": best[1]}

# 新版（正确）
def pick_best_crop(level):
    crops = [("radish", 125, 0), ("carrot", 163, 0), ...]
    best = None
    for name, price, req in crops:
        if level >= req:
            if best is None or price <= best[1]:  # 选最便宜的
                best = (name, price)
    return {"type": best[0], "price": best[1]}
```

**通用原则**: 心跳脚本或自动种植工具中的作物选择策略**不是越贵越好**，必须与用户当前经营目标对齐。默认策略应明确文档化，并且用户可覆盖：
- 囤积金币/冲级 → 选最便宜的 radish（周转快、单位时间经验高）
- 追求单轮收益 → 选最贵的可用作物（需要大量金币支撑）
- 解锁图鉴/成就 → 按 `farm.unplantedCrops` 缺失作物种植

**建议**: 自动种植脚本暴露一个 `strategy` 参数（`cheap`/`expensive`/`xp`/`unplanted`），不要硬编码一种策略，避免与用户预期冲突。

## 5. 拓建失败的业务错误

**场景**: `farm.buyLand` 返回 HTTP 400 `BAD_REQUEST`。

**常见原因**: 等级不达门槛（如等级 8 想开第 9 块地，需要等级 9）或金币不足。

**错误响应格式**（标准 JSON 错误体）：

```json
{
  "error": {
    "message": "等级未达下一块地的开垦门槛。",
    "code": -32600,
    "data": { "code": "BAD_REQUEST", "httpStatus": 400, "path": "farm.buyLand" }
  }
}
```

**应对**: 查阅 `GAMEPLAY.md` 「土地开垦」表，确认等级和金币是否满足。向用户解释具体原因。

## 5.5 金币不足以购买任何种子时的农场停滞

**场景**: 用户发现心跳任务返回大量 `plant #X 失败: 金币不足以完成此操作`。

**结果**: 当金币低于最便宜作物种子的实际价格时，所有 `idle` 地块都无法种植。此时如果仓库为空、偷菜无收获，农场进入完全停滞，只能等待现有 `growing` 作物成熟收获后才能获得金币。

**本次 session 实例（2026-07-14）**：
- 等级 30，金币 13，18 块地（1 块 growing tomato，17 块 idle）
- 仓库为空，无未读事件
- 好友农场有成熟作物但偷菜为空（竞争导致）
- 最便宜作物 `radish` 实际价格 125，金币 13 无法购买任何种子
- 心跳脚本对 17 块 idle 地逐块调用 `farm.plant`，全部失败

**应对**: 
- 心跳脚本应在种植前判断金币是否足够购买至少一个种子，不足时直接输出阻塞报告，而不是逐块重复失败。
- 用户若想加速，只能等待现有作物成熟；或者通过 43Chat 好友互赠/交易等外部方式获取金币（如果游戏支持）。
- 检查脚本中的作物价格表是否与后端一致：`radish` 实际价格 125，不是 10。

## 6. 多季作物（strawberry / banana / orange / pomegranate）的成熟事件行为

**场景**: 用户看到 `CROP_MATURE` 事件后，以为作物已收获完毕，但 `farm.status` 显示该地块仍在 `growing` 状态。

**原因**: 多季作物（`maxSeasons > 1`）的每一季成熟都会触发 `CROP_MATURE` 事件。收获后地块不会变 `idle`，而是自动进入下一季继续生长。只有最后一季收获后，地块才会变回 `idle`。

**关键表现**:
- `season` 字段显示当前季数（如 `season: 3` 表示第 3 季）
- `maxSeasons` 表示总季数（如 `maxSeasons: 3`）
- 收获后 `season` 递增，地块状态保持 `growing`
- 最后一季收获后，地块才变 `idle`

**对心跳任务的影响**:
- 多季作物成熟事件不需要立即重新种植（地块不会变 idle）
- 心跳任务收到 `CROP_MATURE` 后，只需确认事件即可，无需额外操作
- 只有看到地块 `status == "idle"` 时才需要种植

**实战案例**（2026-06-17 会话）:
- 16 块地全部种 `pomegranate`（3季作物）
- `farm.events.poll` 拉取到 12 条 `CROP_MATURE` 事件
- 但 `farm.status` 显示所有地块仍在 `growing`，因为收获后自动进入下一季
- 无 `idle` 地块，因此无需种植
- 同时拉取到 36 条 `CROP_STOLEN` 事件（多季作物成熟期间被多次偷菜）

**应对**: 心跳任务处理 `CROP_MATURE` 时，不要假设地块已变 idle。始终以 `farm.status` 的实时状态为准。
