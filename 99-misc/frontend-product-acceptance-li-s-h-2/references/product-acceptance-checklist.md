# 前端产品验收清单

`frontend-product-acceptance` 以产品经理、业务负责人和最终用户视角，判断已开发功能是否满足 PRD、用户需求和真实业务流程。

## 1. 职责边界

产品验收不做代码审查，也不重复做浏览器 QA 取证。它判断：功能是否满足 PRD，页面信息是否足够业务人员使用，流程是否符合实际业务逻辑，状态权限和数据是否正确流转，用户是否能更少出错地完成工作。

## 2. 输入依据

按顺序读取：

1. `docs/product/PRD.md` 或项目约定 PRD。
2. 用户本轮需求和已确认变更。
3. `frontend-browser-qa` 的视觉截图目录（如有，目标项目 `docs/qa-screenshots/`）、流程记录、Playwright/MCP DOM 断言或失败记录。
4. 必要的 API/schema/DTO/Mock、字段模型、状态枚举和权限说明。
5. 用户反馈截图或问题描述。

缺少 PRD 时通常结论为“依据不足”，并交给 `frontend-product-docs`。缺少浏览器证据时先调用 `frontend-browser-qa`，不要重复截同一批图。

## 3. 与 browser QA 分工

`frontend-browser-qa` 负责取证：页面能否打开、布局是否异常、表格/弹窗/滚动/响应式是否可用、动态点击输入是否跑通、视觉截图和 DOM 断言是否覆盖关键状态。

`frontend-product-acceptance` 负责判断：这些页面状态是否满足 PRD，字段和按钮是否对目标用户有用，流程是否符合业务，状态和数据是否闭环。

## 4. 验收维度

功能完整性：PRD 要求是否都能完成；入口是否自然；主流程是否闭环；不做范围是否被误实现；是否减少人工判断、重复输入或跨页面查找。

用户角色和信息呈现：目标角色最关心什么；页面是否优先展示决策必须信息；列表、详情、弹窗和确认页是否足够支撑操作；是否隐藏无关信息。

行业软件判断方向：仓库看数量、批次、差异、库位和异常；采购看供应商、订单、到货、价格和交期；财务看金额、结算、发票、付款和成本；审核看来源、变更、风险、差异和审批依据。这些不是固定业务规则。

业务流程和状态：状态流转是否符合 PRD；是否跳过必要状态；终态是否只读；审核、确认、删除、作废、导出等高风险动作是否有依据和二次确认；失败、取消、重复提交、无权限、接口异常是否有处理。

数据流转：新增、编辑、提交后是否正确回显；列表、详情、编辑弹窗、确认弹窗数据是否一致；用户输入、系统生成、审计字段、计算字段是否边界清晰；字段单位、枚举、状态文案、金额口径、数量口径是否符合 PRD。

## 5. 结论和分流

- `通过`：核心功能、业务流程、角色信息、状态和数据流转均符合 PRD，浏览器 QA 证据充分。
- `有条件通过`：主流程可用，但存在不阻塞交付的小问题或待确认项。
- `不通过`：核心功能、业务流程、权限、状态或数据流转不符合 PRD 或真实业务需要。
- `依据不足`：缺少 PRD、浏览器 QA 证据、业务规则或关键截图。

| 类型 | 交给谁处理 | 沉淀位置 |
| --- | --- | --- |
| PRD 缺口 | `frontend-product-docs` | 目标项目 `docs/product/PRD.md` |
| 业务逻辑问题（状态、权限、字段含义、流程等） | `frontend-fix-loop` 修复后沉淀到目标项目 PRD | 目标项目 `docs/product/PRD.md` |
| 显示/交互证据不足 | `frontend-browser-qa` | skill 长期维护文件 |
| 实现问题（显示/代码/代码质量） | `frontend-fix-loop` 修复后沉淀到 skill 文件 | skill 长期维护文件 |
| 契约问题 | `frontend-doc-sync` | skill 长期维护文件 |

## 6. 输出格式

```text
产品验收结论：通过 / 有条件通过 / 不通过 / 依据不足

读取依据：
- PRD：docs/product/PRD.md
- 浏览器 QA：<截图目录或记录>
- 其他：<需求/API/用户反馈>

角色视角：
- <角色>：需要看到什么、能做什么、是否满足

功能和流程检查：
- 功能完整性：<结论>
- 业务流程：<结论>
- 状态流转：<结论>
- 数据流转：<结论>
- 权限和高风险动作：<结论>

问题清单：
| 类型 | 位置 | 当前表现 | PRD/业务依据 | 影响 | 建议处理 |
| --- | --- | --- | --- | --- | --- |

待确认：
- <需要用户或产品确认的问题>

后续动作：
- <调用哪个 skill 或修改哪个文档/代码>
```

## 7. 规范缺口沉淀

发现本文件规则缺失或表达不清时，更新本文件（`frontend-product-acceptance/references/product-acceptance-checklist.md`）。

无论触发场景是 fix-loop 沉淀、用户直接指出还是 Agent 自发现，都按 `frontend-fix-loop/references/fix-loop.md` 第6节"文档更新指引"执行：全文搜索→判断有无相似规则→强化/改写/新增→输出沉淀文件和动作。
